为什么使用pod

Pod,是kubernetes项目中最小的API对象。什么意思呢，就是kubernetes项目中调度的最小单位是pod，一个pod中有一个或者多个容器。
我们都知道docker容器的三大技术原理是“namespace做隔离，Cgroup做限制，rootfs做文件系统”，那为什么kubernetes项目又要设计出pod这个全新概念呢？原因还是在于容器的本质。
很多人刚开始都会拿容器和虚拟机来做比较，而容器的本质是进程，一个容器就是一个进程，所以容器与虚拟机并不在同一个维度里。而在linux操作系统里，进程都不是单独运行的，都是以进程组的方式来相互协作来运行一个程序。我们可以使用pstree命令来看看系统中正在运行的进程的树状结构。

可以看到，每个程序都是以进程组的形式在运行。
而在kubernetes中，其实就是将进程与进程组的概念映射到了容器与pod的技术中。当然我们也有这样考虑，说我们可以在容器里设置affinity的约束，让相关的容器运行在同一个主机上，这样不是一样可以运行一个程序吗？ 那我们假设kubernetes集群里有两个节点：node1上有3G可用内存，node2上有2.5G可用内存。这时我们要运行一个rsyslogd的程序，就要分别制作rsyslogd的三个进程的容器，然后不得不让main、imklog、imuxsock三个容器运行在同一个节点上，所以必须在另两个容器设置affinity=main的约束，他们三个容器设置的内存都为1G。然后顺序执行“docker run main”“docker run imklog”“docker run imuxsock”创建三个容器。那么假设有一种情况是main和imklog都被调度到了node2上，这时node2的可用内存只有0.5G了，不够运行imuxsock容器了，可是根据affinity=main的约束，imuxsock又只能运行在node2上。但是如果这三个容器组成一个pod，kubernetes直接调度pod的话，自然就选择node1了，而不会考虑node2了。

一个pod里的所有容器是共享一些资源的，这个在kubernetes里又是怎么做的呢？
答案是在kubernetes里，pod的实现是通过一个infra的中间容器，在一个pod中，都是先创建一个infra容器，其他用户定义的容器，都是通过join infra容器的 Namespace的方式与infra容器相关联。Infra容器的镜像就是 k8s.gcr.io/pause,这个镜像在kubernetes初始化的时候就下载了下来，一个节点上有多少个pod，相应的就有多少个infra容器。

对于volume来说，kubernetes也是把所有volume的定义设计在pod层，一个volume对应宿主机上的目录只有一个，pod里的容器都声明挂载这个volume，那么这些容器之间就是共享这个volume对应的宿主机目录。例如下面这个例子：

为了便于理解，我们是不是可以这样理解容器呢：容器相当于一个进程，容器的镜像相当于”.exe”安装包，pod才是相当于传统的虚拟机，kubernetes就相当于宿主机操作系统呢。
凡是调度、网络、存储以及安全相关的属性，基本上都是pod级别的。

Lifecycle

定义容器状态发生变化时触发的“钩子”。比如在容器启动时，执行一个操作，或者容器被杀死时，执行某个操作等。

在kubernetes里，如果要设置hosts文件内容，一定要通过这种方法。如果直接修改了hosts文件，在pod被删除重建后，kubelet会自动覆盖掉被修改的内容。

imagePullPolicy

默认值是always,即每次创建pod都重新拉取一次镜像，如果定义为 never或ifNotPresent,则pod永远不会主动拉取镜像，或者只有宿主机上不存在这个镜像时才拉取。

总结

Pod是kubernetes中最小的调度单位，是kubernetes中“编排”的一种思想，我们不可能把一个虚拟机里的应用无缝的迁移到一个容器中。而相比虚拟机，容器的性能优势必然伴随着其无法模拟本地物理环境的缺陷，所以我们在把应用要搬到kubernetes中时，必然要考虑这个应用的所有进程，pod才算是能扮演虚拟机的角色。
